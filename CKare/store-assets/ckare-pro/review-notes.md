# CKare Pro reviewer notes

## Executable local-demo path

No production reviewer account is required for this build. The app opens in a fictional, signed-in local-demo state; Profile can toggle that state without contacting an authentication service.

1. Home shows fictional Jordan Ellis data and a shared-policy blocker for payer enrollment.
2. Profile → Complete demo payer review removes that blocker and enables covered-work acceptance.
3. Learn → Complete next module updates progress; View preparation track opens the local preparation state. Course completion is not a license.
4. Work supports local accept or decline decisions. After acceptance, the checklist advances to documentation submitted.
5. Earnings → View documentation-needed claim opens fictional claim detail. Claim acceptance does not guarantee payment.
6. Profile can queue a local support request and toggle local sign-in state.
7. Profile → Delete account first shows explicit confirmation. Confirming ends at “Deletion request submitted and pending”; it never claims deletion completed.

All names, bookings, credentials, and claims are fictional and remain on device for this demo session.

## Owner-controlled production integrations

Before production submission, the owner must connect and validate authentication, persistent profile and learning storage, credential and payer sources, booking/visit services, claims and payout systems, support delivery, privacy links, and server-side deletion processing. Reviewer credentials, if later required, belong in the store console—not this repository.

## Important boundaries

- CKare Pro is not a licensing authority and does not issue licenses.
- Insurance verification and claim acceptance do not guarantee payment.
- CKare is not emergency dispatch; immediate danger belongs with 911 or local emergency services.
- The local support and deletion actions are request states, not completed backend operations.
